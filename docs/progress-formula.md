# 진척도 산정 공식

## 공식

`progress = docs(0-20) + skeleton(0-30) + features(0-30) + alpha(0-20) = 0-100`

## 단계별 정의

### docs (0-20)

PRD/스펙/ADR/Implementation plan 등의 문서화 완성도.

| 점수 | 기준 |
| --- | --- |
| 0-4 | README 한두 줄, 또는 이미지 위주 |
| 5-9 | README가 컨셉 + 실행법 + 폴더 구조 설명 |
| 10-15 | PRD 또는 Implementation Spec 1개 + 컨셉 문서 1개 |
| 16-20 | PRD + ADR + 로드맵 + 운영규약(CLAUDE.md) + 벤치마크 분석 등 다층 문서 |

### skeleton (0-30)

기능 구현을 떠받치는 **하네스 엔지니어링과 서브에이전트 구성의 완성도**. 단순 빌드/실행 골격을 넘어, 검증 하네스(`harness/check.sh`·`npm run ci`), 에이전트 작업 분담(서브에이전트·역할 정의), 자동 리뷰/플레이테스트 프로토콜이 얼마나 갖춰졌는지 본다.

| 점수 | 기준 |
| --- | --- |
| 0-6 | 빈 리포 또는 실행 골격만, 하네스·에이전트 구성 없음 |
| 7-14 | 빌드/실행 골격 + 디렉토리 구조, 검증 하네스는 초기 단계 |
| 15-22 | 검증 하네스(`check.sh`/CI) 동작 + 서브에이전트 역할 분담 정의 |
| 23-30 | 하네스 + 서브에이전트 구성 + 자동 리뷰/합성 플레이테스트 프로토콜까지 정착 |

### features (0-30)

PRD에 정의된 핵심 기능의 실제 동작 비율.

| 점수 | 기준 |
| --- | --- |
| 0-5 | 기능 거의 미구현, 랜딩/데모 페이지만 |
| 6-12 | 핵심 기능 1-2개가 mock data로 동작 |
| 13-19 | PRD 절반의 기능이 실 데이터로 동작 |
| 20-25 | PRD 80% 기능 동작, 일부 엣지 케이스 미해결 |
| 26-30 | PRD 100% + UX 폴리시 진행 중 |

### alpha (0-20)

알파/베타 등급 검증, 사용자 테스트, 폴리시.

| 점수 | 기준 |
| --- | --- |
| 0-2 | 내부 검증 미실시 |
| 3-7 | 합성 플레이테스트 또는 셀프 QA 보고서 |
| 8-13 | 알파 사용자 테스트 N=5+, 보고서 작성 |
| 14-18 | 알파 N=10+ 종합 보고서, 핵심 버그 수정 완료 |
| 19-20 | 클로즈드 베타 진입 가능 등급 |

## 점수 변경 가이드

- 사용자 또는 검증 가능한 외부 신호(GitHub release, TestFlight 등록, 합성 플레이테스트 보고서)에 근거할 것
- 한 번에 5점 이상 점프 시 근거 명시
- 알파 점수는 외부 사용자 N명 이상이 실제 테스트했을 때만 부여
- **공식 개정 이력** — docs는 0-25→0-20, skeleton은 0-25→0-30으로 조정됐고 skeleton 정의가 "하네스·서브에이전트 완성도"로 바뀌었다. 기존 skeleton 점수는 새 기준으로 `/coach`·`/refresh` 시 재평가 대상이다.
