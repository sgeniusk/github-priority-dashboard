# /new-project — 새 프로젝트 킥오프 어시스턴트

사용자가 만들고 싶은 것을 함께 구상하고, 결정되면 **하네스 엔지니어링을 고려한 첫 프롬프트**까지 작성한다. 대시보드에 머무르지 않고 실제 프로젝트 착수를 돕는 단계다.

## 1단계 — 유형 파악

먼저 무엇을 만들고 싶은지 묻는다. `$ARGUMENTS`에 힌트가 있으면 활용한다. 유형은 보통 넷 중 하나다.
- **게임** — 웹/네이티브 게임
- **앱** — 생산성/도구 앱
- **웹사이트/콘텐츠** — 랜딩·문서·교육 사이트
- **기타**

## 2단계 — 아이디어 구상 (대화)

한 번에 결론 내지 말고 질문으로 함께 좁힌다.

**게임이면** — 장르, 코어 게임플레이 루프(플레이어가 반복하는 30초~3분 행동), 세션 길이, 타깃 플랫폼(웹/모바일), 닮고 싶은 레퍼런스 게임, 차별점, 최소 스코프(MVP에서 뺄 것)를 묻는다. sgeniusk의 기존 게임(삼국지 시즈폴, 전지적 군주 시점, Mandate of Heroes, AI 컴퍼니 타이쿤)과 겹치지 않는지 확인한다.

**웹사이트/콘텐츠면** — 목적, 닮고 싶은 사이트를 묻는다. 이때 사내 레퍼런스로 **`ai-builder-school`**(Next.js 15 + React 19 + MDX + Tailwind, 콘텐츠 데이터 모델 + 무결성 검증)을 명시적으로 제시하고, 그 구조를 따를지 다른 사이트를 참고할지 함께 정한다.

**앱이면** — 핵심 사용자 시나리오 1개, 데이터 모델, 인증 필요 여부, 웹/네이티브, MVP 슬라이스를 묻는다.

아이디어가 막연하면 구체적 후보 2~3개를 제시해 고르게 한다.

## 3단계 — 하네스 엔지니어링 첫 프롬프트 작성

방향이 정해지면, 새 리포에서 바로 쓸 **첫 프롬프트**를 코드 블록으로 출력한다. sgeniusk의 기존 프로젝트 관행을 반영한다.

첫 프롬프트에 반드시 포함할 것
- **PRD 우선** — 코드 전에 `docs/PRD.md`를 먼저 쓰게 한다. 핵심 기능, 비기능 요구사항, Acceptance Criteria 포함
- **하네스(harness)** — `harness/check.sh`(또는 `npm run ci`) 같은 검증 스크립트를 골격과 함께 만들게 한다. 빌드·린트·테스트·핵심 동작 검증을 한 커맨드로
- **디렉토리 골격** — `src/`, `docs/`, `harness/` 등 초기 구조 명시
- **MILESTONE_PLAN** — MVP를 3~5개 마일스톤으로 쪼개고 각 Exit 조건 정의
- **AGENT_REVIEW_PROTOCOL** — 에이전트가 산출물을 자가 검증하는 체크리스트
- 게임이면 **합성 플레이테스트 프로토콜**(가상 플레이어 N명 시나리오)도 포함

## 4단계 — 대시보드 등록 (선택)

마지막에 이 프로젝트를 `projects.json`에 추가할지 묻는다.
- 추가한다면 `projects.schema.md`의 필드 구조를 따른다.
- `tool` 태그(claude/codex/hermes/hybrid)는 **반드시 사용자에게 물어서 정한다.** 자동 지정 금지.
- `progress`는 착수 전이므로 docs/skeleton/features/alpha 모두 낮게(또는 0) 시작하고, `status`는 `active`, `sprint`는 사용자와 상의해 배정한다.
- `rank`는 기존 항목과 우선순위를 비교해 사용자와 함께 정한다.
